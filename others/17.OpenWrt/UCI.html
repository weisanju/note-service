openWrt Basic Configuration

The UCI system

  1. The abbreviation UCI stands for Unified Configuration Interface,(统一配置接口)，它是一种将OpenWrt服务的配置集中管理的系统。

  2. UCI是OpenWrt White Russian系列中基于NVRAM的配置的继任者。它是主要的配置用户界面，用于最重要的系统设置，包括主网络接口配置、无线设置、日志功能和远程访问配置。

  3. OpenWrt软件仓库中的许多软件包已与UCI系统兼容。只需根据相应的UCI文件中选择的设置编写原始配置文件（由程序读取），即可使应用程序与UCI兼容。这是在运行/etc/init.d/中的初始化脚本时完成的。有关更多信息，请参阅初始化脚本。因此，当使用此类UCI兼容的初始化脚本启动守护程序时，您应该意识到程序的原始配置文件将被覆盖。例如，在Samba/CIFS的场景下，运行/etc/init.d/samba start时，文件/etc/samba/smb.conf将被UCI配置文件/etc/config/samba中的UCI设置覆盖。

  4. 此外，应用程序的配置文件通常存储在RAM中，而不是闪存中，因为它不需要存储在非易失性存储器中，并且根据UCI文件的每次更改进行重写。如果您希望调整原始配置文件中UCI不可用的设置，有办法禁用UCI，例如在cifs.server中，您可以看到如何禁用Samba的UCI。

  5. 这段内容指出了一些不兼容UCI的程序，提供了一份方便的非UCI配置文件列表，用户可以查看并进行相关设置。但对于大多数第三方程序，应该查阅其自身的文档以获取更准确的配置信息。

Common principles

OpenWrt的中央配置被分成几个文件，这些文件位于/etc/config/目录中。每个文件大致对应于它配置的系统部分。您可以使用文本编辑器编辑配置文件，也可以使用命令行实用程序uci修改它们。UCI配置文件也可以通过各种编程API（如Shell、Lua和C）进行修改，这也是Web界面（如LuCI）对UCI文件进行更改的方式。

init.d reload UCI

在更改UCI配置文件时，无论是通过文本编辑器还是命令行，受影响的服务或可执行文件必须通过init.d%20by%20an-,init.d%20call,-%2C%20such%20that%20the)调用(重新)启动(或在某些情况下仅重新加载)，以便将更新的UCI配置应用于它们。许多程序通过简单地使它们的init.d脚本首先基于/etc/config中更新的UCI配置 更新它们的标准程序特定配置文件，然后重新启动可执行文件来与UCI兼容。这意味着仅直接(重新)启动可执行文件，而不调用适当的init.d脚本，将无法按预期行事，因为它尚未将配置更新合并到程序的标准配置文件中。

假设你想将设备的IP地址从默认的192.168.1.1更改为192.168.2.1。要做到这一点，修改 /etc/config/network 为 option ipaddr 192.168.2.1

然后重启 ：/etc/init.d/network restart

File syntax

UCI（Unified Configuration Interface）配置文件的结构和语法。 UCI配置文件通常由一个或多个config语句组成，这些语句被称为“sections”，每个section又包含一个或多个option语句来定义实际的值。 在UCI配置文件中，以#开头的行被视为注释，即使在字符串字面值之外的#也是如此。如果一行中包含#字符，那么#及其后面的所有字符都被视为注释并被忽略。 下面是一个简单配置文件的示例，可以参考uci_dataobject_model了解更多信息。

config(sections) -> options

package 'example'
 
config 'example' 'test'
        option   'string'      'some value'
        option   'boolean'     '1'
        list     'collection'  'first item'
        list     'collection'  'second item'
  1. config 'example' 'test' 语句定义了一个类型为example、名称为test的部分的开始。还可以有匿名部分，只有类型，没有名称标识符。类型对于处理程序来决定如何处理 enclosed options 很重要。
  2. option 'string' 'some value'和选项'boolean' '1'行在该部分中定义了简单的值。请注意，文本选项和布尔选项之间没有语法差异。按照惯例，布尔选项可以具有以下值之一来指定假值：'0'、'no'、'off'、'false'或'disabled'；或者具有以下值之一来指定真值：'1'、'yes'、'on'、'true'或'enabled'。
  3. 在以列表关键字开头的行中定义了一个具有多个值的选项。所有具有相同名称的列表语句，例如我们的示例中的collection，将被合并为一个具有与配置文件中相同顺序的值的列表。在我们的示例中，collection选项将具有两个值：'first item'和'second item'。
  4. 选项和列表语句的缩进是一种约定，用于提高配置文件的可读性，但在语法上并不是必需的。
  5. 如果一个选项在使用时没有被指定，并且不是必需的，系统会默认使用一个值。但如果这个选项在使用时没有被指定且是必需的，可能会导致应用程序出错或产生其他不希望的行为。
  6. 一种禁用配置部分的方法：当一个配置部分没有可禁用的选项时，可以通过将配置部分标识符（或类型，在这个例子中）重命名为不被使用这些值的进程识别的值来禁用它。通常，将配置部分类型/标识符命名为"disabled_identifier"就足够了
  7. 使用引号的情况。一般情况下，不需要在标识符或值周围使用引号。只有当所包含的值中包含空格或制表符时，才需要使用引号。此外，在输入配置选项时，使用双引号代替单引号也是合法的。
  8. 这段内容解释了在使用UCI标识符和配置文件名时的字符限制。UCI标识符和配置文件名只能包含小写字母a到z、数字0到9和下划线（_），不允许使用连字符（-）。而选项值可以包含任何字符，只要它们被正确引用（即用引号括起来）。

Command-line utility

这段内容讲述了在OpenWrt中，通常需要直接更改UCI配置文件来调整设置。但是，为了脚本编写的目的，也可以使用uci命令行工具来读取和更改UCI配置。对于需要自动解析UCI配置的开发人员来说，使用awk和grep解析OpenWrt的配置文件是多余、不明智和低效的。uci实用程序提供了所有与修改和解析UCI相关的功能。

这段内容讲述了使用uci编写配置文件时的注意事项。使用uci编写配置文件时，文件总是被整体重写，不被识别的命令会被省略，包括注释等多余的行也会被删除。如果你想保留自己的注释和空行，就不应该使用命令行工具，而应该正常编辑文件。需要注意的是，一些文件在安装应用程序时已经包含了很多注释，例如uHTTPd配置文件。此外，一些应用程序如LuCI也使用uci实用程序，因此可能会重写UCI配置文件。

这段内容解释了在配置文件中，如果存在多个相同类型的部分，UCI（Unified Configuration Interface）支持对它们进行类似数组的引用。例如，在/etc/config/system中定义了8个NTP服务器，UCI允许你通过system.@timeserver[0]引用第一个服务器，通过system.@timeserver[7]引用最后一个服务器。你还可以使用负索引，比如system.@timeserver[-1]表示最后一个服务器，system.@timeserver[-2]表示倒数第二个服务器，依此类推。这在向列表末尾追加新规则时非常方便。下面是一些示例。
